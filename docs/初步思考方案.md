鸿蒙开发问题定位系统设计方案文档

1. 文档信息

文档名称：鸿蒙开发问题定位系统技术设计方案
版本：v1.0
实现语言：Python
前端方案：方案 A，FastAPI + Jinja2 模板 + 原生 JavaScript
运行模式：单机本地部署
目标阶段：技术验证 Demo
并发能力：固定并发 3
Agent 工具：支持本地 CLI 方式接入 Claude Code、OpenCode，后续可扩展其他 CLI Agent

⸻

2. 系统目标

本系统服务于鸿蒙开发者的问题定位场景。用户提交以下输入后，系统自动调用本机上的 CLI Agent 工具，结合 Skill 执行问题分析，并以任务方式展示执行过程、中间结果和最终结论。

支持的输入包括：
	1.	问题描述
字符串形式，可能是自然语言，也可能直接包含报错信息、错误码、错误文本
	2.	hilog 日志
支持文本输入和文件上传两种形式
	3.	问题代码片段
用户认为存在问题的位置，但不保证是真正根因位置

系统要求：
	1.	后端通过 Python 实现
	2.	前端采用方案 A
	3.	后端通过本地脚本调用 CLI Agent
	4.	所有中间结果和最终结果落盘到本地文件系统
	5.	多用户管理先采用本地目录隔离方式
	6.	支持实时查看任务进展
	7.	初始并发数固定为 3
	8.	全局配置可通过配置文件统一管理
	9.	支持配置本地代码仓路径、本地文档仓路径、可用 Agent 工具列表等全局变量

⸻

3. 设计原则

3.1 快速验证优先

本系统当前阶段定位为技术验证，不追求复杂分布式能力，优先保证链路完整、执行稳定、过程可见。

3.2 本地化闭环

输入、任务、过程、日志、结果全部本地化落盘，保证每个任务都可回溯、可复盘、可调试。

3.3 任务隔离

每个任务使用独立工作目录，避免并发执行时文件污染和上下文混乱。

3.4 Agent 可替换

业务层不直接依赖 Claude Code 或 OpenCode 的具体命令格式，而是通过统一执行接口进行调用。

3.5 Skill 过程结构化

Skill 执行过程中的每个阶段都必须产出结构化中间文件，便于前端实时展示、后端复盘和后续评估。

3.6 配置集中管理

所有路径、并发限制、支持的 Agent、默认 Skill、超时等参数统一由配置文件管理。

⸻

4. 系统总体架构

4.1 总体架构说明

系统采用单体服务架构，主要包含以下模块：
	1.	Web 展示层
负责问题提交、任务列表、任务详情、中间结果和最终结果展示
	2.	API 服务层
负责接收请求、保存输入、创建任务、返回任务状态和结果
	3.	任务管理层
负责任务创建、状态流转、队列管理、执行调度
	4.	Agent 执行层
负责封装 Claude Code、OpenCode 等 CLI Agent 的调用方式
	5.	Skill 运行层
负责组织 Skill 输入、生成工作区文件、写入中间结果、解析输出结果
	6.	本地存储层
负责所有用户数据、任务数据、输入文件、中间产物、日志和结果文件的落盘管理

5. 业务流程设计

5.1 主流程

flowchart TD
    A[用户提交问题] --> B[保存原始输入]
    B --> C[创建任务记录]
    C --> D[生成工作目录]
    D --> E[生成标准化 Skill 输入]
    E --> F[任务入队]
    F --> G{当前运行中任务数小于3}
    G -->|是| H[启动 CLI Agent]
    G -->|否| I[等待调度]
    H --> J[Skill 执行]
    J --> K[持续写入中间结果]
    K --> L[前端轮询展示]
    J --> M[生成最终结果]
    M --> N[任务完成]

5.2 任务生命周期

任务状态枚举如下：
	•	created
已创建，尚未入队
	•	queued
已入队，等待执行
	•	running
正在执行
	•	succeeded
执行成功并生成最终结果
	•	failed
执行失败
	•	cancelled
已取消
	•	timeout
超时终止

任务阶段枚举如下：
	•	input_preprocess
	•	intent_analysis
	•	log_analysis
	•	code_analysis
	•	hypothesis_build

11. Agent 适配层设计

11.1 统一执行接口

所有本地 CLI Agent 必须通过统一接口调用。

from abc import ABC, abstractmethod
from pathlib import Path

class AgentExecutor(ABC):
    @abstractmethod
    async def execute(self, task_id: str, workspace: Path, skill_name: str, input_file: str) -> int:
        pass

    @abstractmethod
    def supports(self, agent_type: str) -> bool:
        pass

11.4 Agent 工厂

根据 agent_type 选择具体执行器。

class AgentExecutorFactory:
    def __init__(self, executors):
        self.executors = executors

    def get_executor(self, agent_type: str):
        for executor in self.executors:
            if executor.supports(agent_type):
                return executor
        raise ValueError(f"Unsupported agent type: {agent_type}")

12. Task 调度与并发设计

12.1 调度模型

系统采用单进程内异步调度模型：
	1.	使用 asyncio.Queue 保存待执行任务
	2.	使用 asyncio.Semaphore(3) 控制最大并发数
	3.	使用后台 worker 循环持续调度任务
	4.	每个任务独立调用本地子进程执行 CLI Agent

⸻

12.2 核心调度逻辑
	1.	新任务创建后写入任务目录
	2.	状态更新为 queued
	3.	放入任务队列
	4.	调度器发现有空闲槽位后启动任务
	5.	状态更新为 running
	6.	执行结束后根据退出码和结果文件决定最终状态

⸻

12.3 超时机制

配置项：runtime.task_timeout_seconds

策略：
	1.	每个任务启动后记录开始时间
	2.	调度器周期性检查心跳和运行时长
	3.	超过配置超时秒数后：
	•	终止子进程
	•	状态更新为 timeout
	•	记录事件
	•	写入错误信息

13. Skill 运行层设计

13.1 Skill 运行目标

Skill 运行层负责：
	1.	生成 Skill 标准输入
	2.	组织任务工作目录
	3.	驱动 Agent 按阶段执行
	4.	将阶段结果写入 artifacts
	5.	输出最终结构化结果

⸻

13.2 Skill 执行阶段定义

阶段 1：输入预处理

提取错误码、错误信息、API 名称、关键字、上下文信息

阶段 2：问题意图识别

归类问题属于权限、参数、生命周期、调用顺序、线程、配置、设备能力等类型

阶段 3：日志分析

分析 hilog 中的错误关键词、调用链、模块、时间顺序和首个异常点

阶段 4：代码分析

分析用户提供片段中的 API 调用、参数、调用上下文和可能误判点

阶段 5：根因假设生成

输出多个根因候选及置信度

阶段 6：证据链合并

将描述、日志、代码中的证据聚合成结构化链路

阶段 7：最终结果生成

生成 final_result.json 和 final_report.md

⸻

13.3 Skill 输出约束

Skill 执行必须满足以下约束：
	1.	每个阶段至少生成一个 JSON 文件
	2.	最终必须生成 output/final_result.json
	3.	最终结果必须是结构化 JSON
	4.	中间结果禁止仅以自然语言纯文本存在
	5.	所有路径必须使用任务工作目录相对路径

17. 异常处理设计

17.1 CLI Agent 不存在

处理方式：
	1.	任务状态置为 failed
	2.	记录 error_message
	3.	向事件流写入 ERROR
	4.	stderr 保存错误详情

⸻

17.2 Agent 返回非零退出码

处理方式：
	1.	标记任务失败
	2.	读取 stderr 作为错误信息补充
	3.	若有部分 artifacts 已生成，仍保留以便复盘

⸻

17.3 最终结果缺失

处理方式：
	1.	即使 CLI 退出码为 0，只要 final_result.json 缺失，任务仍判定为 failed
	2.	错误原因写为 final_result_missing

⸻

17.4 任务超时

处理方式：
	1.	杀死子进程
	2.	任务状态改为 timeout
	3.	事件流写入超时事件

⸻

17.5 上传文件超限

处理方式：
	1.	拒绝创建任务
	2.	接口返回 400
	3.	明确提示文件超出大小限制
