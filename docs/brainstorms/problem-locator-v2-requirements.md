# Problem Locator V2: Adaptive Diagnostic Pipeline

## Problem Frame

HarmonyOS 开发者在遇到应用崩溃、API 调用异常、错误码等故障时，需要一个自动化诊断工具，能够综合利用知识库、官方文档和代码仓库快速定位根因并给出修复建议。

当前 problem-locator skill 存在以下核心问题：
1. 代码探索触发条件模糊，对所有问题类型采用相同的探索深度，导致简单问题浪费 token、复杂问题探索不足
2. 知识库查询策略单一，未充分利用 `api_chain.json` 的调用链追踪能力
3. 完整的官方文档仓（166 个错误码文档 + 模块开发指南）未被有效利用
4. 缺乏模块识别机制，无法精准定位应查询哪个模块的知识库和文档

## Requirements

**诊断策略**

- R1. 采用两阶段分级诊断策略：分诊（Triage）→ 深潜（Deep Dive）。分诊阶段快速匹配知识库和文档，若置信度不足则进入深潜阶段做深度分析（包括深度知识库追踪、文档查询和代码级探索）
- R2. 三种问题类型均需覆盖：错误码驱动型、崩溃/冻屏日志型、API 调用异常型
- R3. 内置模块映射表，包含以下子映射，按问题类型提供不同的信号识别路径：
  - 错误码前缀 → 模块名（适用于错误码驱动型）
  - DOMAIN 标识 → 代码仓目录（适用于崩溃/冻屏日志型）
  - `.so` 库名 → 模块名（适用于崩溃日志调用栈）
  - API 名称前缀 → 模块名（适用于 API 调用异常型）
  - hilog domain_id → 模块名（适用于 hilog 日志）
  - 模块名 → 文档仓开发指南目录路径
  - 注意：d.ts 声明名、文档仓 kit 名、代码仓目录名、开发指南目录名之间无机械对应关系，每个子映射需独立维护

**分诊阶段（Triage）**

- R4. 日志解析：从输入中提取错误码、系统事件名、DOMAIN、调用栈（含 .so 库名）、模块标签、API 名称、hilog domain_id 等结构化线索
- R5. 知识库查询：按识别的模块，优先查询 `error_codes.json`（错误码精确匹配）和 `api_chain.json`（按已知 API 名精确匹配调用链），其次查询 `common_issues.md`（已知问题模式匹配）
- R6. 文档仓查询：使用混合策略定位错误码文档：
  - 优先：若模块映射表中包含该模块的 errorcode 文件路径映射，直接读取
  - 兜底：在 `docs_repo_root/zh-cn/application-dev/reference/` 下 Grep 搜索错误码数字，定位到对应的 errorcode 文件
  - 不假设 errorcode 文件路径可通过命名规则推导（同一 kit 下可有数十个 errorcode 文件，错误码前缀可跨 kit 出现）
- R7. 置信度评估与阶段路由（此处的"置信度"是阶段路由的二值判断，与 R12 中根因候选的"诊断置信度"是不同概念）：
  - **高置信度（直接输出）**：模块特定错误码精确匹配（如 6600101 对应 av_session），且知识库或文档提供了充分的排查信息
  - **低置信度（进入深潜）**：仅有通用错误码（如 201/202/401，跨所有模块通用）、仅有模糊的问题描述、或知识库和文档均未提供足够信息
  - 边界情况：通用错误码 + 明确的模块上下文 → 按高置信度处理
  - 注意：高置信度仅表示"找到了匹配的结构化信息"，不保证该信息能解决用户的具体问题。R12 中的根因候选诊断置信度（高/中/低三档）由实现阶段定义判定标准

**深潜阶段（Deep Dive）**

- R8. API 链路追踪：利用 `api_chain.json` 按调用栈中的 C++ 函数名反向追踪 JS API → NAPI 桥接函数 → C++ 实现函数的完整调用路径，定位实际出错位置。注意：此能力仅对已构建知识库的模块有效（当前 2 个模块）；对未覆盖模块，深潜阶段以代码仓直接搜索（R9）为主要机制
- R9. 代码仓搜索：根据调用栈信息在 `code_repo_root` 中搜索相关实现代码，分析错误处理逻辑。搜索范围限定在 R3 识别的模块目录内，避免全仓扫描
- R10. 文档深度查询：通过模块映射表定位开发指南目录（`docs_repo_root/zh-cn/application-dev/{guide-dir}/`），搜索 API 使用指南和最佳实践；查询 `faqs/` 目录下的常见问题
- R11. 证据交叉验证：将知识库匹配结果、文档参考、代码分析进行交叉验证，构建可信的证据链。当不同来源的信息矛盾时，以代码仓实际实现为准

**诊断输出**

- R12. 输出结构化 JSON 结果到 `output/final_result.json`（通过 Write 工具），包含：问题摘要、提取线索、知识库匹配项、根因候选列表（含诊断置信度和证据链）、修复建议。后端 `_parse_result()` 改为直接读取 `final_result.json` 文件（当文件不存在时回退到 stdout raw_output）
- R12a. 当信号（错误码前缀、DOMAIN、.so 名等）不匹配映射表中任何条目时，在输出中明确标注"模块未识别"，使映射缺口可观测而非静默降级
- R13. 每个根因候选附带证据来源标注（知识库 / 文档 / 代码），便于开发者溯源验证

**知识库覆盖范围说明**

- V2 的价值在知识库覆盖范围内外有所不同：
  - **已覆盖模块**（当前 2 个）：分诊阶段可快速通过知识库匹配解决问题，两阶段分级策略充分发挥价值
  - **未覆盖模块**：分诊阶段主要依赖文档仓查询（这是 V1 缺失的核心能力），V2 的主要价值来自文档仓的集成利用而非分级策略本身。随着知识库覆盖扩大，分级策略的价值将逐步提升

## Success Criteria

- 对于知识库已覆盖的模块，模块特定错误码驱动型问题能在分诊阶段直接匹配并输出结果
- 崩溃/冻屏日志能通过 .so 库名 → 模块映射 + API 链路分析定位到具体的 C++ 实现函数
- API 调用异常问题能通过文档仓的 API 参考和开发指南给出排查方向
- 当知识库未覆盖相关模块时，系统通过文档仓 Grep 搜索 + 代码仓搜索仍能给出有意义的诊断结果
- 后端能正确解析 skill 输出的结构化 JSON（不再依赖 stdout JSON 解析）

## Scope Boundaries

- 不实现自动化的知识库内容更新（诊断结果不会自动回写到知识库）
- 不支持多问题并发的交互式诊断（一次只分析一个问题描述）
- 不实现用户对诊断流程的实时干预能力（如中途调整方向）
- 不包含性能基准测试或诊断准确率评估体系（后续独立建设）
- 模块映射表为手动维护，不实现自动发现或同步机制

## Key Decisions

- **两阶段分级策略**：针对不同问题复杂度采用不同探索深度，避免一刀切的 token 消耗。核心价值：简单问题快速匹配节省 token，复杂问题有足够的探索深度
- **映射表 + 搜索的混合策略**：模块识别使用内置映射表（按问题类型提供不同信号路径），文档定位使用映射表优先 + Grep 兜底。不假设命名体系之间存在可推导关系
- **知识库 → 文档仓 → 代码仓的三级信息源**：按信息密度和查询成本排序，优先使用结构化的知识库数据，其次查文档，最后才做代码探索。sdk_repo_root 作为深潜阶段的补充信息源
- **输出机制改为文件读取**：修复后端 _parse_result 从 stdout 解析 JSON 的不可靠机制，改为直接读取 skill 写入的 final_result.json 文件（当文件不存在时回退到 stdout raw_output）

## Dependencies / Assumptions

- 知识库的文件格式遵循 knowledge-builder skill 产出的 8 文件规范
- `code_repo_root`、`docs_repo_root`、`sdk_repo_root` 三个路径在 config.yaml 中正确配置且可访问
- `sdk_repo_root` 中的 `.d.ts` 声明文件用于深潜阶段确认 API 参数和权限要求
- 模块映射表中的子映射需要在首次实现时根据代码仓和文档仓的实际结构填充并验证
- 三个外部仓库的内容应与开发者使用的 HarmonyOS 版本对应。仓库内容过期会导致诊断结果引用错误的文件路径或过时的 API 签名，此为已知的不可检测的失败模式
