# HarmonyOS 开发者问题辅助定位系统

## Problem Frame

鸿蒙开发者在开发过程中遇到问题（报错、崩溃、异常行为）时，需要手动在多个代码仓中查找原因。当前有 100+ 个部件实现代码仓、API 声明仓和文档仓，但缺乏一个统一的辅助诊断工具。

已有系统 api_dfx_2.0 解决的是**主动预防**问题（批量扫描存量代码的规范性），但无法帮助开发者**被动响应**已遇到的具体问题。

本系统定位为轻量验证平台，核心目标是验证"Agent + Skill 能否有效辅助开发者定位鸿蒙问题"，而非构建生产级基础设施。

## Requirements

**平台层 — 轻量验证框架**

*用户界面*

- R1. 提供 Web 界面，支持开发者提交问题描述、故障日志（hilog、HiviewDFX crash/freeze 日志等）、代码片段作为诊断输入。日志格式不限定为单一类型，系统需能处理多种鸿蒙故障日志格式
- R3. 前端采用混合风格 — 主页为任务列表/看板视图，点击进入后为对话式分析过程展示，能实时呈现 Agent 的分析步骤、中间产物和最终结论

*任务管理*

- R2. 支持两种任务类型：知识库构建任务、问题定位任务，通过统一界面操作

*基础设施*

- R4. 每次任务执行的结构化输入、过程数据、中间产物、最终结果均落盘到本地文件系统，支持事后复盘和 Skill 迭代优化
- R5. 全局配置（代码仓路径、文档仓路径、Agent 工具选择、并发参数等）通过配置文件统一管理，禁止散落在业务代码中

**Skill 层 — 知识库构建模式**

- R6. 支持对指定模块/部件进行主动探索，生成结构化的知识文件（JSON/Markdown），覆盖：错误码定义与实现映射、API 声明与实现链路、常见问题模式与解决方案、关键代码路径摘要
- R7. 知识库构建支持单模块手动触发执行，产出的知识文件按模块组织，供后续问题定位时优先查询。批量执行和定时调度作为验证通过后的优化方向
- R8. 知识库内容支持覆盖式重建（重新探索已有模块时完整覆盖）。增量追加能力作为验证通过后的优化方向

**Skill 层 — 问题定位模式**

- R9. 接收开发者的问题输入后，**优先查询知识库**中已有的相关信息（按错误码、API 名、模块名等关键词匹配），给出初步定位结果
- R10. 当知识库信息不足以回答时，**回退到实时代码探索** — 根据问题描述和故障日志中的线索（错误码、系统事件名如 THREAD_BLOCK_6S、DOMAIN 标识、调用栈中的库/函数名），在具体实现代码仓中搜索和定位根因
- R11. 最终输出结构化诊断结果，包含：按相关性排序的根因候选列表、证据链、修复建议、参考的知识库条目或代码位置。置信度评分方式（Agent 自评或规则计算）留待规划阶段确定
- R12. 问题定位过程中持续输出中间步骤和中间产物，供前端实时展示分析进展

**资源集成**

- R13. 通过配置引用 api_dfx 下的代码仓（DataBases）、API 声明仓（interface_sdk-js）和文档仓资源路径，不硬编码绝对路径
- R14. 可复用 api_dfx_2.0 中已有的 NAPI 映射模式知识、API 提取脚本等已有能力，通过配置或 Skill 引用集成

## Success Criteria

- **问题定位验证（核心）**：对至少 2 个不同类型的问题（如冻屏问题 THREAD_BLOCK_6S + 错误码类问题如 201 权限）完成辅助定位，输出结构化诊断结果。纯实时分析路径必须先验证通过
- **知识库构建验证（第二步）**：对一个示例模块（如 base_location）完成知识库构建，产出结构化知识文件，并对比有/无知识库时问题定位的效果差异
- 开发者能在 Web 界面中看到完整的分析过程（知识库查询 → 若知识不足则代码探索 → 根因分析 → 修复建议）
- Skill 的输入、中间产物、最终结果均有本地文件可复盘，便于迭代优化 Skill 的分析逻辑
- 修改 Skill 定义后无需修改平台代码，仅需重新执行任务即可验证效果

## Scope Boundaries

- 不构建生产级系统（分布式部署、高可用、用户认证等均不在范围内）
- 不构建通用知识库引擎（使用结构化文件 + 文件搜索，不引入向量数据库或搜索引擎）
- 不替代 api_dfx_2.0 的审计能力（两者为独立功能，可共享数据但各有侧重）
- 当前阶段不追求 Agent 工具的广泛适配（优先 Claude Code CLI，验证通过后再考虑扩展）
- 不实现自动化的 CI/CD 集成或持续知识库更新流水线（手动触发即可）

## Key Decisions

- **独立项目 vs 扩展 api_dfx_2.0**：选择在 location_vod 中独立构建。原因是功能定位不同（辅助诊断 vs 合规审计），独立项目便于专注验证 Skill 效果，不受 api_dfx_2.0 现有架构约束
- **渐进式验证策略**：验证分两步走。**第一步**先验证问题定位 Skill 的纯实时分析能力（Agent 能否直接从代码仓中定位问题），**第二步**再引入知识库构建模式，对比有/无知识库时的诊断效果。避免在未验证 Agent 基本分析能力的情况下就投入知识库构建工作
- **结构化文件作为知识库载体**：选择 JSON/Markdown 文件而非数据库。理由是与 Agent 的工作方式天然契合（直接 Read/Grep），无需额外基础设施，验证阶段最务实
- **混合前端风格**：选择看板 + 对话式分析的结合。验证阶段看板可简化为基本任务列表，重点放在对话式分析过程的展示
- **双模式 Skill**：知识库构建和问题定位拆分为两个独立 Skill。问题定位 Skill 优先实现（验证阶段核心），知识库构建 Skill 作为第二步
- **api_dfx_2.0 复用方式**：优先复用 api_dfx_2.0 已有的知识资产（NAPI 映射模式 MAPPING_PATTERNS.md、kit-api-extract 产出的 api.jsonl/impl_api.jsonl）作为知识库的种子数据或 Skill 参考材料，而非在新 Skill 中从零重新实现探索逻辑

## Dependencies / Assumptions

- 假设 Claude Code CLI 已安装且可用，作为主要的 Agent 执行引擎。CLI 命令名为 `claude`（非 `claude-code`），Skill 通过 `claude -p "/skill_name\nparams..."` 方式调用，Skill 文件需放在 `.claude/skills/` 目录下
- 假设 api_dfx/DataBases 下的代码仓目录结构保持稳定（当前约 105 个目录）
- 假设 api_dfx/api/interface_sdk-js 中的 API 声明文件格式（.d.ts）保持稳定
- **[已部分验证] 开发者提交的故障日志格式多样** — 包括 hilog、HiviewDFX crash/freeze 日志等，信息粒度差异大。已有一个冻屏案例（THREAD_BLOCK_6S），日志中包含 Reason、DOMAIN、STRINGID、调用栈、进程信息等线索。需要在规划前收集更多样本（至少包含错误码类、崩溃类、冻屏类），确认各类日志的可解析字段和诊断入口
- 假设开发者在提交问题时能提供至少一种有效输入（问题描述、hilog 或代码片段）
